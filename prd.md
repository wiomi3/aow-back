## PRD: Система управления календарем мероприятий

### 1\. Обзор продукта

Веб-приложение для публикации и администрирования расписания мероприятий.

**Стек технологий:**

- **Frontend:** React, Vite, Tailwind CSS, shadcn/ui.
- **Backend:** Hono (vercel).
- **База данных:** PostgreSQL (через Prisma ORM).
- **Авторизация:** better-auth.

### 3\. Функциональные требования (Frontend)

**3.1. Клиентская часть (Публичная)**

- **Сетка месяца:** Полноэкранный календарь (месяц). В ячейках дней отображаются "чипсы" или цветные блоки событий. Цвет блока берется из EventType.color.
- **Детали события:** При клике на событие открывается модальное окно (или Drawer из shadcn), где отображаются: название, описание, время начала/конца, площадка (адрес) и список задействованных сотрудников.
- _Фильтры не требуются._

**3.2. Панель Администратора**

- **Авторизация:** Страница входа (/admin/login), обрабатываемая клиентом better-auth.
- **Управление календарем:**
  - Клик по пустой дате в календаре открывает форму создания нового события.
  - Клик по существующему событию открывает форму редактирования/удаления.
  - Форма события включает: выбор дат и времени (startAt, endAt), селекты для выбора типа события и локации, а также Multi-select для выбора сотрудников.

- **Управление справочниками (Dictionaries):**
  - Отдельные страницы или табы для управления зависимыми данными.
  - **Локации:** Добавление/редактирование названия и адреса.
  - **Типы событий:** Название и цвет (color picker для UI).
  - **Типы сотрудников:** Например, "Вокалисты", "Аниматоры" и т.д.
  - **Сотрудники:** Имя и привязка к типу сотрудника.

### 4\. Бизнес-логика и защита от накладок (Backend)

Hono-бэкенд должен не только записывать данные, но и следить за целостностью расписания.

**Алгоритм защиты от двойного бронирования сотрудников:**

При создании (POST) или обновлении (PUT) события со списком employeeIds:

1.  Бэкенд делает запрос к БД, проверяя, есть ли у этих сотрудников другие события, где:
    - Новое startAt < Существующее endAt
    - Новое endAt > Существующее startAt

2.  Если пересечение найдено, Hono возвращает ошибку 409 Conflict с сообщением вида: _"Сотрудник \[Имя\] уже занят в событии \[Название\] с 14:00 до 16:00"_.
3.  Транзакция отменяется, событие не создается.

### 5\. Архитектура API (Hono Endpoints)

Все /api/admin/\* маршруты защищены middleware от better-auth, проверяющим сессию администратора.

#### Публичные маршруты

- GET /api/events — Получение событий. Принимает query-параметры start и end (ISO даты первого и последнего дня месяца), чтобы не грузить из базы все события за год.

#### Защищенные маршруты (Админ)

**События:**

- POST /api/admin/events — Создать (с проверкой накладок).
- PUT /api/admin/events/:id — Обновить.
- DELETE /api/admin/events/:id — Удалить.

**Справочники (CRUD для каждого):**

- GET/POST/PUT/DELETE /api/admin/locations
- GET/POST/PUT/DELETE /api/admin/event-types
- GET/POST/PUT/DELETE /api/admin/employees
- GET/POST/PUT/DELETE /api/admin/employee-types

### 6\. Требования к UI-компонентам (shadcn/ui)

- **Формы:** Использование react-hook-form в связке с zod для валидации на клиенте (особенно проверка, что endAt позже, чем startAt).
- **Календарь:** Стандартный компонент Calendar из shadcn отлично подходит для выбора дат в формах (Date Picker). Для **самой сетки месяца** (где отображаются события) потребуется кастомный компонент на базе CSS Grid (7 колонок), так как стандартный shadcn Calendar не предназначен для вывода массива событий внутри ячеек.
- **Модалки:** Компоненты Dialog (для десктопа) или Sheet (выплывающая панель сбоку) для создания/редактирования сущностей без перехода на новые страницы.
